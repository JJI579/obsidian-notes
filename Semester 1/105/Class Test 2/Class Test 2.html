Takeaways

Lazy Evaluation

it will evaluate outermost function

ghci> x = (head [], error "error") 
ghci> y = fst x `div` (2 - 2) 
ghci> y

with this, it will evaluate (2-2) and identify divide by 0 before computing head []

IO Unboxing

When computing values from an IO function, it has to be unboxed, example

tester :: IO Int
tester = do
    x = 2
    return x

x <- tester
show 2*x

if not unboxed, then itll throw an error

Deriving

Ensures that custom classes and types derive what is required, example Read

data Weather = Hot | Warm | Cold | Freezing deriving (Show, Eq, Ord)

read "Hot" :: Weather
-- This will not work ^^ !!!

-- When will it work?
-- If it is like Below

data Weather = Hot | Warm | Cold | Freezing deriving (Show, Eq, Ord, Read)
read "Hot" :: Weather
-- Now it'll work due to it being derived!

Order of Constructors

The order of constructors are as such

lowest heirarchy | middle | top-most

so in the scenario of a constructor

data Weather = Hot | Warm | Cold | Freezing deriving (Show, Eq, Ord, Read)

Hot < Warm < Cold < Freezing

-- So in the Scenario of 
Warm >= Hot
-- This is true since Warm is greater than Hot
